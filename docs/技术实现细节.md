# 项目技术实现细节

本文档记录 `obsidian-canvas-card-actions` 当前代码结构和关键实现，重点覆盖本轮落地后的“快速动作 + 预览工作台”交互架构。

## 1. 项目结构

项目采用分层架构，核心目录如下：

### 核心入口

- `src/main.ts`
- `src/settings/CanvasCardActionsSettingTab.ts`
- `src/settings/ICanvasCardActionsSettings.ts`

### 适配器层 `src/adapters/`

- `CanvasAdapter.ts`：封装 Canvas 数据读写、节点查找和保存
- `ClipboardAdapter.ts`：封装剪贴板写入和提示
- `StorageAdapter.ts`：封装插件设置读取、保存和旧字段迁移
- `VaultAdapter.ts`：封装 Markdown 文稿创建

### 领域层 `src/domain/`

- `models/`：`Badge`、`CanvasData` 等领域模型
- `strategies/`：`PositionSortStrategy`、`BadgeSortStrategy`

### 服务层 `src/services/`

- `CardService.ts`：卡片拆分与创建
- `BadgeService.ts`：徽章读取、写入和回填
- `ContentService.ts`：选区快照、排序和文本拼装
- `MergeService.ts`：复制、工作台、新建卡片、新建文稿等输出编排
- `PreviewWorkbenchService.ts`：工作台一次性状态管理

### 表现层 `src/presentation/`

- `commands/`：右键菜单命令和命令面板命令
- `modals/`：拆分弹窗、徽章弹窗、旧的拖拽排序弹窗
- `views/`：预览工作台 `ItemView`
- `styles/`：样式注入

## 2. 当前核心能力

插件当前主要包含四组能力：

- 单卡片复制
- 多卡片快速复制 / 快速拼合 / 工作台预览
- 卡片拆分
- 徽章和属性管理

其中多卡片复制与合并现在共享统一的内容生成链路。

## 3. 多卡片交互架构

### 3.1 右键菜单

单卡片右键菜单保持：

- `复制卡片内容`
- `拆分卡片...`
- `添加/编辑徽章`
- `管理卡片属性`

多选右键菜单已经收敛为：

- `一键复制`
- `一键拼合`
- `打开预览...`
- `管理卡片属性`

### 3.2 命令面板

为了兼容快捷键和重度用户，命令面板仍然保留旧链路：

- 合并为新卡片
- 在工作台中预览
- 合并为新文稿
- 手动排序拼合

同时增加了新的快捷入口：

- 将当前选区一键复制
- 将当前选区一键拼合
- 打开预览工作台

## 4. 统一的数据流

当前多卡片链路已经统一为：

1. 从 Canvas 读取选区
2. 生成选区快照
3. 按排序模式得到有序卡片列表
4. 根据需要构建最终文本
5. 输出到剪贴板、Canvas 卡片、工作台或 Markdown 文稿

### 4.1 为什么要引入快照

引入快照的主要目的是把工作台从 live selection 中解耦：

- 工作台可以作为一次性任务流独立存在
- 拖出独立窗口后仍能继续工作
- 用户后续修改白板选区不会污染已经打开的工作台

### 4.2 快照内容

快照最少包含：

- 节点 `id`
- 文本内容
- 坐标和尺寸
- 徽章及其类型

## 5. 排序实现

排序仍然采用策略模式：

- `PositionSortStrategy`
- `BadgeSortStrategy`

在此基础上，新增了手动排序状态：

- `manualOrderIds`

工作台里的排序行为是：

- `位置`：自动排序，只读
- `徽章`：自动排序，只读
- `手动`：以当前排序结果为起点，再允许拖拽微调

这意味着“手动”不再是独立输出链路，而是排序层的一种模式。

## 6. 内容生成实现

`src/services/ContentService.ts` 负责内容生成。

### 6.1 关键方法

- `createSelectionSnapshot`
- `getOrderedCards`
- `buildMergedContent`
- `copyMergedContent`

### 6.2 设计特点

- 输入既可以是 live selection，也可以是快照
- 先返回“有序卡片列表”，再决定是否生成字符串
- 徽章模式下支持输出 `[徽章] 内容`
- 手动模式通过 `manualOrderIds` 排序，而不是复制整份文本

## 7. 工作台实现

工作台基于 Obsidian 原生 `ItemView` 实现，主文件是：

- `src/presentation/views/MergeWorkbenchView.ts`

兼容导出文件是：

- `src/presentation/views/MergePreviewView.ts`

### 7.1 视图布局

当前工作台分三块：

1. 顶部工具栏：排序模式、来源 Canvas、卡片数量
2. 中部顺序列表：展示当前顺序，手动模式可拖拽
3. 底部预览和输出区：折叠预览 + 三个输出按钮

### 7.2 状态来源

工作台不持久化状态，只把状态保存在 view 实例中。状态结构定义在：

- `src/types/WorkbenchState.ts`

由：

- `src/services/PreviewWorkbenchService.ts`

进行更新和重算。

### 7.3 预览性能策略

当前预览区采用以下策略减轻大内容场景的开销：

- 默认折叠
- 展开后才计算合并文本
- 渲染增加 `200ms` 防抖
- 卡片数超过阈值时显示延迟渲染提示

## 8. 输出实现

### 8.1 剪贴板

通过 `ClipboardAdapter.writeTextWithNotice` 统一处理，保证复制成功后给出一致提示。

### 8.2 新建 Canvas 卡片

`MergeService` 会：

1. 从当前排序结果构建文本
2. 根据快照中的第一张卡片确定锚点
3. 在右侧偏移创建新卡片
4. 调用 `CanvasAdapter.addNode` 和 `requestSave`

### 8.3 新建 Markdown 文稿

通过 `VaultAdapter.createMergedDocument` 创建文件：

- 目录：当前 `.canvas` 同目录
- 文件名：`<canvas basename>-卡片合并-时间戳.md`
- 路径冲突时自动避让

### 8.4 自动恢复源 Canvas

工作台里的“新建卡片”不再依赖当前激活 leaf，而是依赖 `canvasFilePath`：

1. 先找已经打开的对应 Canvas leaf
2. 找不到时自动打开对应 `.canvas`
3. 拿到 Canvas 实例后再写入节点

这解决了工作台被拖出独立窗口后无法落回原 Canvas 的问题。

## 9. 设置项实现

当前设置项包括：

- `canvasCardDelimiter`
- `sortPriority`
- `enableBadges`
- `defaultSortMode`

### 9.1 旧字段迁移

旧字段 `mergeDefaultOrder` 已被迁移为 `defaultSortMode`。

迁移逻辑在 `StorageAdapter.loadSettings()` 中完成，保证旧用户升级后不丢配置。

### 9.2 设置页语义

设置页文案也已同步为：

- `一键排序方式`

它影响：

- 一键复制
- 一键拼合
- 打开工作台时的默认排序模式

## 10. 兼容与遗留结构

当前仍保留以下遗留能力用于兼容：

- `DragSortModal`
- `CopyByPositionCommand`
- `CopyByBadgeOrderCommand`
- `CopyByManualOrderCommand`
- 旧的 merge 命令类

这些能力已经不再挂到主右键菜单，但仍可作为过渡层存在，后续可视实际使用情况继续清理。

5. 手动拖拽后执行 `复制 / 新建卡片 / 新建文稿`
6. 关闭原 Canvas 后从工作台新建卡片
